# 系统架构设计

## 一句话定位

这是一个“制造业品牌 GEO 现状审计系统”：前端负责配置客户、问题库、模型和查看报告；后端负责批量采样、数据归档、结构化评估和报告生成。

## 系统边界

本项目是 **GEO 效果检测 / 审计系统**，不是 GEO 内容生产系统。

当前服务器上存在两套独立系统：

- `geoflow`
  - 目录：`/opt/geoflow`
  - 技术栈：`Laravel + Docker`
  - 用途：GEO 内容生产 / 流程系统
- `manufacturing-geo-audit`
  - 目录：`/opt/manufacturing-geo-audit`
  - 技术栈：`Python 标准库 + SQLite + 静态前端`
  - 用途：GEO 效果检测 / 审计系统

结论：

- 两套系统必须独立部署
- 禁止把本项目覆盖到 `/opt/geoflow`
- 文档、仓库、服务名和端口都应单独维护

## 总体架构

```text
前端看板
  ├─ 客户与品牌档案
  ├─ 问题库管理
  ├─ 模型与采样任务配置
  ├─ 运行监控
  ├─ 结果分析
  └─ GEO 报告

后端服务
  ├─ 项目/客户管理
  ├─ 问题库服务
  ├─ 模型适配器
  ├─ 采样任务调度器
  ├─ 回答归档服务
  ├─ 结构化评估服务
  ├─ 指标聚合服务
  └─ 报告生成服务

数据层
  ├─ SQLite / PostgreSQL
  ├─ 原始回答表
  ├─ 引用来源表
  ├─ 评估结果表
  └─ 报告快照表
```

## 后端设计

### 1. 客户与品牌模块

保存每个测试项目的基础信息。

核心数据：

- 客户名称
- 品牌名
- 公司全称
- 产品品类
- 目标地区
- 目标市场
- 官网域名
- 竞品列表
- 重点产品页
- 资质、案例、参数资料

作用：

- 给问题库变量填充提供基础数据。
- 给评估服务判断“是否提到目标品牌 / 竞品 / 自有官网”提供依据。

### 2. 问题库模块

管理问题模板和实际问题。

建议分两层：

- `question_templates`：通用模板，例如“采购某品类有哪些品牌值得了解？”
- `questions`：套入客户品牌、品类、地区后的实际问题。

问题分类：

- 品牌直问
- 品类推荐
- 采购决策
- 技术参数
- 竞品对比
- 售后风险
- 区域供应商
- 海外采购

### 3. 模型适配器模块

每个模型服务商单独做 adapter，不要把所有 API 调用写在一个文件里。

建议适配器：

- OpenAI
- Anthropic
- Google Gemini
- Perplexity
- DeepSeek
- Kimi
- 豆包
- 通义

每个 adapter 统一输出：

```json
{
  "provider": "openai",
  "model": "gpt-4.1",
  "search_enabled": false,
  "response_text": "...",
  "citations": [],
  "usage": {},
  "latency_ms": 1234,
  "raw_response": {}
}
```

这样后续分析服务不用关心不同 API 的返回格式。

### 4. 采样任务调度模块

负责把“客户 × 问题 × 模型 × 模式 × 重复次数”展开成任务队列。

采样维度：

- 客户品牌
- 问题
- 模型
- 是否联网
- 地区
- 语言
- 重复次数
- 时间批次

初版可以不用复杂队列，直接用同步批处理：

```text
for project in projects
  for question in questions
    for model in models
      for repeat in repeats
        call_model()
        save_result()
```

后续再升级为任务队列：

- Celery / RQ
- Redis
- 后台 worker
- 失败重试
- 限流控制

### 5. 原始回答归档模块

这是系统的证据层，必须完整保留。

保存内容：

- 请求参数
- 问题正文
- 模型名与版本
- 是否联网
- 回答原文
- 引用来源
- token 用量
- 成本估算
- 请求时间
- 错误信息

原则：

分析可以重算，但原始回答不能丢。

### 6. 结构化评估模块

把原始回答转成可统计指标。

初版可以规则 + LLM 混合：

- 规则判断品牌是否出现。
- 规则判断竞品是否出现。
- 规则抽取引用域名。
- LLM 辅助判断推荐强度、情绪倾向、事实错误。
- 人工复核高风险结果。

核心指标：

- 目标品牌是否出现
- 目标品牌排名
- 推荐强度
- 竞品共现
- 是否引用客户官网
- 是否引用竞品官网
- 是否存在事实错误
- 风险等级

### 7. 报告生成模块

把指标聚合成客户能看懂的结论。

输出：

- Markdown 报告
- HTML 报告
- CSV 明细表
- 后续可扩展 PDF

报告结构：

- 总体结论
- 模型对比
- 品牌可见度
- 竞品共现
- 引用来源
- 错误风险
- 内容资产缺口
- 30 天优化建议

## 前端设计

前端不要先做复杂大屏，先做一个工作台。

### 1. 项目首页

用途：查看所有客户 GEO 审计项目。

核心组件：

- 项目列表
- 客户名称
- 品牌名称
- 最近运行时间
- 品牌命中率
- 高风险错误数
- 报告状态

### 2. 客户档案页

用途：配置客户基础资料。

字段：

- 品牌名
- 公司全称
- 官网域名
- 产品品类
- 目标地区
- 主要竞品
- 重点页面
- 备注资料

### 3. 问题库页

用途：维护问题模板和本项目实际问题。

功能：

- 导入 CSV
- 新增问题
- 按问题类型筛选
- 预览变量替换结果
- 标记优先级

### 4. 采样任务页

用途：配置并启动批量测试。

功能：

- 选择模型
- 选择是否联网
- 设置重复次数
- 设置 temperature
- 选择问题范围
- 预估任务数量
- 预估成本
- 启动任务

### 5. 运行监控页

用途：看采样执行状态。

核心信息：

- 总任务数
- 成功数
- 失败数
- 当前模型
- API 错误
- 平均耗时
- 预估成本

### 6. 结果分析页

用途：看结构化结果。

核心视图：

- 分模型品牌命中率
- 分问题类型命中率
- 目标品牌平均排名
- 竞品共现排行
- 引用来源域名排行
- 事实错误列表
- 原始回答抽屉

### 7. GEO 报告页

用途：生成和查看客户报告。

功能：

- 自动生成报告
- 人工编辑结论
- 导出 Markdown / HTML
- 查看明细证据

## 前后端技术建议

### 初版推荐

```text
前端：Next.js / React
后端：Python FastAPI
数据库：SQLite
任务执行：Python 批处理脚本
分析：Pandas
报告：Markdown + HTML
```

原因：

- 初期重点是跑通采样和分析，不是复杂工程。
- SQLite 足够承载早期客户项目。
- Python 更适合 API 调用、数据处理和报告生成。
- 前端可以后置，先用简单工作台展示结果。

### 第二阶段

```text
数据库：PostgreSQL
队列：Redis + RQ / Celery
前端：Next.js
后端：FastAPI
部署：Docker Compose
```

适用条件：

- 多客户并发。
- 多批次定时复测。
- 采样任务耗时明显增加。
- 需要账号和权限。

## MVP 开发顺序

### 第 1 步：命令行 MVP

先不做前端。

完成：

- 读取 CSV 问题库
- 配置 2-3 个模型
- 批量调用 API
- 保存原始回答
- 生成结果 CSV

### 第 2 步：分析 MVP

完成：

- 品牌命中判断
- 竞品共现统计
- 引用域名统计
- 简单报告 Markdown

### 第 3 步：前端工作台

完成：

- 项目列表
- 问题库导入
- 任务启动
- 结果表格
- 报告预览

### 第 4 步：产品化

完成：

- 账号权限
- 队列执行
- 定时复测
- 报告版本管理
- 客户交付视图

## 关键设计原则

1. 原始回答优先保存，分析结果可以后算。
2. API 基准测试和联网搜索测试分开，不混成一个结论。
3. 系统提示词和产品排序只能做行为采样，不声称拿到了内部规则。
4. 前端先服务工作流，不做装饰型大屏。
5. 报告要落到内容资产缺口，而不是只给模型跑分。

## 最小可用版本范围

MVP 不需要马上实现所有模型。

建议首批只接：

- OpenAI：纯模型 + 搜索模式
- Perplexity：搜索型结果
- DeepSeek 或 Kimi：中文模型对照

首批只做 30-50 个问题，每题每模型重复 3 次。这样就可以产出第一份可用 GEO 现状报告。

# 路径可行性评估

## 结论

整体可行，适合做成一个面向制造业客户的 GEO 现状审计工具。它能稳定解决三个问题：

- 客户品牌在不同模型回答中是否被提到。
- 被提到时，位置、语气、事实描述和推荐理由是什么。
- 未被提到时，模型更倾向推荐哪些竞品或信息来源。

但需要修正一个关键假设：API 无状态调用可以减少上下文污染，但不能直接等同于“真实用户搜索结果”。真实产品端经常有联网搜索、浏览器检索、用户画像、地区、语言、时间、工具路由和产品层规则。报告里应明确区分“API 基准测试”和“产品端搜索体验测试”。

## 可行的原因

1. 问题库可以标准化。

同一批问题在不同模型中重复执行，能够形成可比较数据。

2. API 请求可以保持低上下文污染。

每个问题单独调用，不携带历史对话，可以减少上一轮回答对下一轮的影响。

3. 回答内容可以结构化归档。

问题、模型、回答、时间、参数、引用、品牌命中、竞品命中、错误类型都可以进入表格。

4. 制造业场景适合做 GEO。

制造业客户通常有明确品类、参数、区域、采购场景、渠道和竞品，适合转化成可测问题。

## 主要风险

### 1. API 结果不等于产品端结果

同一个模型，在 API 和网页产品里的回答可能不同。产品端可能自动联网、调用搜索、加入系统提示词、做安全过滤或个性化排序。

建议：报告中保留两个维度：

- API 基准结果。
- 产品端 / 搜索增强结果。

### 2. 模型回答有随机性

即使同一个问题、同一个模型，也可能产生不同回答。

建议：

- temperature 尽量设为 0 或最低。
- 同题至少重复 3 次，用命中率而不是单次结果下结论。
- 记录 seed，如果模型支持。

### 3. 不同模型的联网能力不一致

有些 API 默认不联网，有些需要显式启用搜索工具，有些搜索引用不可控。

建议：

- 给每个模型标记能力类型：纯模型、联网搜索、引用返回、工具调用。
- 不把纯模型结果和联网结果混在一个排行榜里。

### 4. 问题库会决定结论质量

如果问题库只问“某某品牌怎么样”，结果会偏浅。真正有价值的是模拟客户采购路径。

建议问题库至少分 7 类：

- 品牌直问。
- 品类推荐。
- 技术参数咨询。
- 采购决策。
- 竞品对比。
- 售后与风险。
- 区域和渠道。

### 5. 法务和客户沟通边界

模型回答可能包含竞品、错误事实或负面评价。对客户交付时不能把模型回答直接当事实。

建议：

- 报告表述为“模型呈现结果”，不要写成“市场事实”。
- 对负面或错误内容单独标记为“需核验风险”。

## 可以完善的地方

### 1. 增加品牌知识输入对照组

除了裸测，还可以测试“加入客户官网资料 / 产品资料后”的回答变化。

这样能回答一个更商业化的问题：客户补齐哪些公开内容后，模型呈现会改善。

### 2. 增加来源分析

如果模型返回引用来源，要记录域名、页面标题、发布时间和是否为客户自有资产。

核心指标：

- 自有官网引用率。
- 第三方媒体引用率。
- B2B 平台引用率。
- 竞品官网引用率。
- 过期页面引用率。

### 3. 增加竞品共现

制造业客户很关心“我没出现时谁出现了”。

核心指标：

- 竞品出现次数。
- 竞品被推荐排名。
- 竞品推荐理由。
- 竞品引用来源。

### 4. 增加事实错误分类

建议把错误分为：

- 公司名称错误。
- 产品品类错误。
- 参数错误。
- 区域错误。
- 资质错误。
- 客户案例错误。
- 把竞品信息混到本品牌。

### 5. 增加可行动建议

报告不要只说“品牌可见度低”，要落到内容资产建设：

- 哪些问题缺少可被引用的官网页面。
- 哪些产品参数需要结构化页面。
- 哪些行业场景需要案例页。
- 哪些竞品内容正在被模型引用。
- 哪些第三方平台值得补充资料。

## 推荐项目定位

不要把它定位成“模型答案监控工具”，而应定位成：

制造业品牌 GEO 现状审计与内容资产优化系统。

这样更容易和客户预算连接，因为它不是单纯跑数据，而是回答：

- 品牌现在在 AI 搜索里怎么被看见。
- 为什么没有被推荐。
- 哪些内容资产缺失。
- 先补哪几类页面最可能改善。

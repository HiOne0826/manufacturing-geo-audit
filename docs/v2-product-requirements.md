# V2 产品需求文档：可追溯、可恢复、可交付的 GEO 审计工作台

状态：已批准，作为 `codex/V2` 的实现与验收基线  
更新日期：2026-07-11

## 1. 产品目标

系统服务少量内部交付人员，用于为制造业客户生成有证据底稿的 GEO 现状审计。V2 保留“项目、问题库、模型、采样、批次、分析”这些既有概念和主要 URL，不建设内容生产、多租户、客户自助或复杂审批。

核心旅程：

```text
选择/创建项目 → 完善项目档案 → 导入并校验问题 → 检查信息源就绪度
→ 配置并命名批次 → 确认任务规模 → 执行监测 → 暂停/继续/重试
→ 数据质检 → 分析与证据复核 → 导出交付
```

用户可直接进入任一页面，不设置强制向导。V2 的成功不是“页面更丰富”，而是同一份采样工作在切换项目、进程重启、外部服务故障和人工重试后仍可解释、可恢复、可复现。

## 2. 用户与原则

主要用户是内部交付执行者和报告审阅者。首版不区分权限角色；公网仍由现有 Nginx Basic Auth、`APP_PASSWORD` 和 Agent token 保护。

产品原则：

- 项目上下文始终可见，任何操作都不得跨项目串线。
- 业务名称优先展示，机器 ID 保留用于排障与追溯。
- “没有数据”“正在刷新”“请求失败”“部分失败”必须是不同状态。
- 暂停、继续、重试允许重复点击，但不得重复创建同一逻辑任务。
- 报告默认读取 current 结果；历史 attempt 不删除，可显式查看与导出。
- 不允许把跨 provider 的 fallback 冒充为原信息源结果。

## 3. P0 功能需求

### FR-01 项目上下文

- 当前项目同时写入 URL `?project_id=` 和本地持久状态；URL 是刷新与深链接恢复时的优先来源。
- 项目 ID 不存在、已归档或无权访问时，清除无效上下文并提示用户重新选择。
- 切换项目必须清除旧项目的活动批次、分析批次范围、查询缓存和未提交采样表单。
- 从全局任务中心或批次列表打开批次详情时，先把当前项目同步为该批次所属项目。

验收：A 项目有运行批次时切换到 B，页面不能显示、暂停、继续或重试 A 的批次；刷新深链接仍恢复 B。

### FR-02 项目生命周期

- 支持编辑所有现有项目字段和归档/恢复；默认列表不显示已归档项目。
- 硬删除入口放在危险操作区，先返回关联问题、批次、运行和评估数量，再要求输入项目名称确认。
- 归档不删除历史数据，也不影响历史报告读取；归档项目不能启动新批次。

验收：未完成影响预览与名称确认时，前端不能提交硬删除。

### FR-03 项目准备度

项目概览显示档案完整度、问题数量、已启用模型、信息源健康、活动批次、可分析样本和唯一的“下一步操作”。缺失项可直接跳转到对应页面。

### FR-04 问题导入

- 保留粘贴、CSV、XLSX 三种习惯。
- 提交前展示合法、重复、跳过和错误行，逐行给出原因。
- 只有用户确认后才写入；重复判断至少覆盖当前项目内标准化后的问题正文。
- 保留原始导入行到 `import_row_json`，便于审计和客户模板回填。

### FR-05 信息源就绪检查

- 启动批次前检查模型启用状态、Key 是否存在、模式能力、联网依赖、worker/队列和熔断状态。
- 阻断项禁止提交，并提供到模型设置或信息源详情的修复入口；告警项允许确认后继续。
- 健康结果不显示 Key、cookie、credential fingerprint 原值或原始响应。

### FR-06 批次业务信息

- 批次增加必填业务名称，以及可选说明、用途、标签。
- 创建时保存不可变配置快照，至少包含项目、问题集合、模型/模式、采样参数和重复次数。
- 批次列表和任务中心优先显示业务名称，机器 ID 作为次要信息。

### FR-07 启动确认与冲突

- 提交前显示项目、问题数、平台/模式、重复次数、总任务数和预计耗时；无法可靠估算时明确写“暂不可估算”。
- 同一项目只允许一个 `queued/running/pause_requested/paused` 活动批次。冲突返回 `409 ACTIVE_BATCH_EXISTS` 和现有批次摘要，不静默复用。
- `client_request_id` 用于幂等；相同 ID 和相同请求返回原批次，相同 ID 与不同请求返回冲突。

### FR-08 生命周期操作

- 暂停含义固定为“停止派发新任务，允许已领取任务完成”；运行中批次先进入 `pause_requested`，在途任务归零后进入 `paused`。
- 继续只派发未完成或符合恢复策略的任务；重复继续不创建重复任务。
- 对同一目标状态重复暂停/继续返回当前状态，不报通用 500。

批次状态：`queued / running / pause_requested / paused / completed / failed_system / cancelled`。兼容读取历史 `failed`，API 序列化时映射为 `failed_system` 或保留 legacy 标识。

批次另返回 `outcome=clean|partial_failure|failure|pending`，把采样失败与调度系统失败分开。

### FR-09 分类重试

- 错误统一归类为 `auth / rate_limit / timeout / region / parse / dependency / unknown`。
- 支持 `scope=all|source|tasks`，提交前展示符合条件的 current 失败任务数。
- 重试产生新 attempt；成功后旧 run 进入历史，新 run 成为 current。
- 现有 `rerun_failed` 作为兼容别名保留，语义等同 `scope=all`。

### FR-10 全局任务中心

- 任一页面都能查看运行、暂停中、系统失败和有采样失败的批次。
- 打开批次时同步项目上下文。
- 轮询失败时保留最后成功数据并提示“任务数据可能已过期”，不能清空成“暂无任务”。

### FR-11 批次详情

展示配置口径、总进度、信息源状态、错误分类、current 运行结果和 attempt history。任一证据行可定位到 task、run 与 attempt；默认不展开原始响应。

### FR-12 分析口径

- 默认展示项目整体 current 结果，可切换到单批次 drilldown。
- 页面明确显示范围、数据截止时间、样本完整度、失败率、批次/配置口径和报告版本。
- 证据样本可跳转到对应批次运行，不将失败样本纳入有效率分母而不作说明。

### FR-13 统一异步状态

所有资源页必须分别处理首次加载、后台刷新、错误、空数据和局部失败；有缓存数据的刷新失败不能覆盖已有内容。

## 4. P1 可信交付

- 数据质检：run 可标记为有效、排除、需复核，并记录理由、操作者和时间。
- 报告版本：草稿、已审核、已冻结、已归档。冻结时保存 summary 与纳入的 run ID 快照，后续重试不改变旧报告。
- 批次对比：比较基线与复测；问题集、模型矩阵、模式或重复次数不同则标记“不可完全同比”。
- 导出中心：客户版/内部诊断版、current/完整 attempt history 明确分开。
- 操作审计：记录批次元数据修改、暂停、继续、重试、项目归档和报告冻结。

## 5. 公共接口契约

- `POST /api/runs/start`：增加 `batch_name`、`description`、`purpose`、`tags`、`client_request_id`；活动批次冲突返回 `409`、`code=ACTIVE_BATCH_EXISTS`、`active_batch`。
- `POST /api/batches/{id}/metadata`：只修改名称、说明、用途和标签，不修改采样配置快照。
- `POST /api/batches/{id}/retry`：接收 `scope`，source 模式接收 source 标识，tasks 模式接收 task ID 列表。
- `GET /api/batches/{id}/attempts`：按 task 返回 append-only attempt 与 current 标识。
- `GET /api/health/live`：只表示 Web 进程存活。
- `GET /api/health/ready`：检查数据库/schema、Redis、worker heartbeat、队列和磁盘可写空间。
- `GET /api/sources/health`：返回 provider/model/mode 维度的静态、主动和被动健康摘要。
- `POST /api/sources/{source}/probe`：触发隔离探针，结果不进入报告样本。
- `GET /api/tasks/active`：返回任务中心所需的跨项目活动与异常批次摘要。
- `GET /api/analytics/summary`：保留已有字段，增加报告版本、配置口径和数据新鲜度。

错误响应统一至少包含 `code`、`message`、`retryable` 和可选 `details`；不得返回凭证、cookie、完整外部请求或响应。

## 6. 验收指标

- 项目切换与深链接 E2E 全部通过，不发生跨项目读写。
- 相同启动、暂停、继续和重试请求重复两次，逻辑 task 数不增加。
- worker 在 claim、外部响应、DB commit 和 RQ ack 前后退出，reconciler 最终能消除永久 running 与 queued 无 job。
- 同一 task 同时只有一个有效 lease；同一逻辑键/generation 同时只有一个 current run。
- 1440、900、390 三档无页面级横向滚动；关键流程可用键盘完成；Lighthouse Accessibility 不低于 95。
- 信息源故障只影响对应 source/mode；跨 provider fallback 必须作为独立 generation 显示。

## 7. 明确不做

P0 不做复杂 RBAC、客户自助、多组织租户、审批流、定期复测、通知和预算管理；SQLite 继续支持本地/单机，正式多 worker 的目标数据库为 PostgreSQL。

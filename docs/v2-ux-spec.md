# V2 UX 规格：保留习惯的工作台改造

状态：已批准，供前端实现与视觉验收使用  
更新日期：2026-07-11

## 1. 体验目标

V2 不改变用户熟悉的“项目 → 问题库 → 采样 → 批次 → 分析”操作语义。改造重点是让项目上下文、任务状态和下一步动作始终明确，并把系统错误从空状态中分离。

视觉保留浅色背景、蓝色主操作和鸵鸟品牌。信息密度面向内部交付人员，不使用大面积营销卡片，不引入新的大型 UI 框架。

## 2. 信息架构与导航

保留现有 URL：

| 分组 | 页面 | URL | 核心任务 |
| --- | --- | --- | --- |
| 工作台 | 总览 | `/` | 系统与当前项目准备度、最近任务 |
| 工作台 | 全部批次 | `/batches` | 跨项目检索和进入批次 |
| 当前项目 | 项目 | `/projects` | 选择、创建、编辑、归档项目 |
| 当前项目 | 问题库 | `/questions` | 导入预览和维护问题 |
| 当前项目 | 采样 | `/sampling` | 配置、预检、确认并启动批次 |
| 当前项目 | 分析 | `/analysis` | 项目整体和批次 drilldown |
| 系统管理 | 模型 | `/models` | provider 配置、能力和探针 |
| 系统管理 | 设置 | `/settings` | 运行环境和安全边界 |

`/batches/:batchId` 是全局可达的详情页。打开时根据批次 `project_id` 同步当前项目，并把 `project_id` 写回 query string。

## 3. 全局外壳

### 侧栏

- 桌面端按“工作台 / 当前项目 / 系统管理”分组，当前页使用颜色、背景和左侧标记三重提示。
- 390px 使用可关闭抽屉；打开后焦点进入导航，关闭后回到触发按钮。
- Logo、产品名和环境标识固定在顶部；退出登录与低频设置放底部。

### ProjectContextBar

- 主内容顶部持续显示当前客户、品牌和品类。
- 切换项目是全局动作：若当前页有未提交表单，弹出确认；确认后清空旧项目局部状态并导航到同类安全页面。
- 无项目时不假装加载失败，显示选择项目和创建项目两个明确动作。

### TaskCenter

- 顶栏显示活动/异常任务数量，点击打开右侧抽屉。
- 条目显示批次业务名称、所属项目、状态、进度和更新时间；机器 ID 使用次级文字。
- 请求失败但有缓存时保留条目并显示“任务数据可能已过期”。

### 全局反馈

- `Toast` 用于已完成且不要求决策的反馈；错误不得只靠 Toast，相关区域内同时显示可恢复动作。
- `ConfirmDialog` 用于离开未保存表单、暂停、重试和归档。
- `DangerConfirmDialog` 用于硬删除，展示影响统计并要求输入项目名。
- 所有 dialog 支持 Escape、焦点圈定和关闭后焦点恢复。

## 4. 页面旅程

### 总览

优先级从上到下：当前项目准备度、唯一下一步操作、活动任务、最近完成批次、信息源异常。系统级指标与项目级指标必须有清晰标签，避免混用。

### 项目

- 列表显示状态、品牌、客户、品类、问题数、最近批次和更新时间。
- 创建/编辑使用同一表单；保存失败保留输入。
- 归档是普通二次确认；硬删除位于详情底部危险区，影响预览未加载完成前不可提交。

### 问题库

导入分三步但不做强制全局向导：选择粘贴/文件 → 本地或服务端预览 → 确认写入。预览摘要显示合法、重复、跳过、错误数量；错误行表格可筛选并展示行号和原因。已存在问题的列表操作保持原入口。

### 采样

采用单页分区：

1. 批次信息：名称、用途、说明、标签。
2. 范围：当前项目和问题数量。
3. 模型矩阵：pure/search 与采样参数。
4. 就绪检查：Key、能力、依赖、worker 和熔断。
5. 任务估算：平台、模式、重复次数、总量、预计耗时。

“启动采样”先打开确认 dialog，不直接提交。阻断项存在时按钮禁用并将焦点引导到第一个阻断项。收到 `ACTIVE_BATCH_EXISTS` 时展示现有批次摘要和“查看现有批次”，不把它显示成未知错误。

### 批次列表与详情

- 列表以业务名称为主，支持项目、状态、日期筛选；进度使用数字和条形图，不仅依赖颜色。
- 详情顶部固定显示业务名称、机器 ID、项目、状态、outcome、创建时间和配置摘要。
- 主要动作按状态出现：暂停、继续、重试、导出；请求 pending 时禁用重复提交，但后端仍必须幂等。
- 失败区先按错误分类和 source 聚合，再按需展开 task；attempt history 使用侧滑详情，不挤压主表。
- `partial_failure` 文案为“已完成，有失败”，`failed_system` 文案为“系统执行失败”，不得共用一个红色“失败”。

### 分析

- 顶部明确“项目整体/单批次”、数据截止时间、有效样本/计划样本和失败率。
- 指标卡、模型对比、问题类型短板、竞品风险、引用结构、证据样本按从结论到证据排列。
- 图表同时提供数值/表格语义；点击证据定位到批次详情的具体 run。
- 请求失败时保留上次报告并显示过期提示；真正无有效样本时显示生成数据的下一步动作。

## 5. 组件状态契约

`AsyncBoundary` 必须接收并区分：

- `initialLoading`：无缓存的首次加载，显示结构化 skeleton。
- `refreshing`：保留内容，仅在标题或控件旁显示轻量进度。
- `error`：显示错误摘要、重试和必要的修复链接。
- `empty`：请求成功但集合为空，显示原因和下一步动作。
- `partial`：部分 source 失败，保留成功内容并标记影响范围。

数据表统一提供 loading、empty、错误、键盘焦点、窄屏横向容器和列优先级。390px 下允许表格自身横向滚动，不允许页面级横向滚动。

状态颜色只做辅助：所有 badge 同时显示文本和图标。`queued/running/pause_requested/paused/completed/failed_system/cancelled` 与 `clean/partial_failure/failure/pending` 分为两组视觉语义。

## 6. 响应式与可访问性

- 1440px：展开侧栏，主内容最大宽度约 1440px；双栏/三栏根据内容密度布局。
- 900px：侧栏可收窄，表单与指标最多双栏，任务中心不遮挡关键操作。
- 390px：抽屉导航，表单单栏，主操作占满可用宽度，触控目标至少 44×44px。
- 使用语义 `nav/main/aside/dialog/table`；页面只有一个 `h1`，标题层级连续。
- 所有输入有可见 label；错误通过 `aria-describedby` 关联；异步结果用克制的 live region 通知。
- 焦点样式不可移除，对比度满足 WCAG AA；关键流程全键盘可完成。

## 7. 前端工程边界与验收

- 保留 React、Vite、React Router、TanStack Query、Zustand、Recharts；按页面懒加载。
- 项目 ID 是 query key 的组成部分；切换项目时取消旧请求并清理旧项目缓存和局部 store。
- 初始 gzip JS 目标低于 150 KB，单个业务 chunk 低于 250 KB；图表只在分析页加载。
- 使用 Vitest、React Testing Library、MSW 覆盖状态组件和项目切换；Playwright 覆盖核心旅程与 1440/900/390 截图；关键页运行 axe。
- Lighthouse Accessibility 目标不低于 95。
